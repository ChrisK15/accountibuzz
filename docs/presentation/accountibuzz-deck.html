Chris Kelamyan & Timothy Do · May 8, 2026
The problem

Most people miss their goals — and it isn't an information problem.

Habit trackers

Habitica, streak apps. Run on the honor system. You are the only audience.

Commitment contracts

StickK, Beeminder. Manufacture stakes with money or anti-charities. Financial, not social.

Neither model captures what actually works in real friend groups — where stakes are social and verification is human.

The opportunity

A small-group, friend-shaped accountability app.

  • Strict-streak mechanic taken seriously — no grace days, no freezes
  • Daily photo or video proof required
  • Verification routes through a single human admin
  • No public discovery, no leaderboard arms race
  • Groups arrive via invite link, not marketing spend

The bet: a tightly-scoped product with one clear opinion lands harder than a feature-rich generalist.

Vision

For people who follow through when friends are watching but not when only they are, Accountibuzz turns a small group into the audience.

Every day, each member submits a short photo or video as proof of their commitment. An admin verifies it. Points and streaks update for everyone to see.

A missed day is visible to the rest of the group within hours — and that visibility is the whole point.

Data model

One unique constraint does most of the work.

-- The canonical race guard for the entire app.
CREATE TABLE submissions (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  group_id    uuid NOT NULL REFERENCES groups(id) ON DELETE CASCADE,
  user_id     uuid NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  local_date  date NOT NULL,            -- derived server-side, never client-sent
  status      text NOT NULL CHECK (status IN ('pending','approved','rejected')),
  media_path  text NOT NULL,            -- private bucket, signed-URL gated
  caption     text CHECK (length(caption) <= 140),
  reviewed_by uuid REFERENCES profiles(id),
  reviewed_at timestamptz,
  created_at  timestamptz DEFAULT now(),

  UNIQUE (group_id, user_id, local_date)  -- ← stops double-submit forever
);
ALTER TABLE submissions ENABLE ROW LEVEL SECURITY;

Two-phase commit: storage upload first, row insert second. Orphans swept by a planned cleanup job.

Security

Row-Level Security is the only authorization boundary.

Boundaries

  • RLS enabled on every public table
  • rls-check.yml CI fails build if disabled
  • service_role never in the mobile bundle
  • anon key treated as public — RLS protects, not secrecy

RPC hardening

  • SECURITY DEFINER + set search_path = public
  • Re-resolve group_id server-side (cross-group attack)
  • Row lock + status guard for double-approve
  • Uniform invite_not_found blocks enumeration

Defense in depth: RLS · RPC check · client check. Any one fails — the other two still hold.

Risks

What could kill it — and what we did about it.

RiskMitigation
Week-2 death spiral
One miss → others disengage → group dies		 Encourage 7–10 invitees for a target of 5–6 · re-engagement prompt in Phase 6 · track group-level rate, not DAU
Admin bottleneck Optimistic pending UI · swipe-stack queue · push pings · planned daily-digest
Strict streak feels punitive Communicate at onboarding · re-entry prompt without violating the rule · hold the line
Push drops silently (Android) notifications_outbox + push-dispatch · EAS dev clients from day one · real-device testing
Upload fails on flaky cellular Two-phase commit · AsyncStorage offline queue · explicit progress UI · no silent success toasts
RLS misconfigured, anon key leaks RLS on by default · rls-check.yml CI gate · pgTAP coverage of every policy
