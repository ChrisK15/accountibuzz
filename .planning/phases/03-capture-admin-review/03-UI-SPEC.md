# Phase 3 — UI Design Contract

> Visual and interaction contract for **Capture & Admin Review** — the core accountability loop. Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Extends `01-UI-SPEC.md` and `02-UI-SPEC.md` — no new visual tokens.** This phase introduces three new full-bleed surfaces (Today / Capture / Admin Review queue), one navigation-shell change (Stack → Tabs), one modification to an existing screen (Group detail gets an admin-only entry row), and **five new RN component primitives** (`GroupCard`, `StatusPill`, `TypeChip`, `QueueBadge`, `SwipeCard` + `CaptureControls`). All inherit the locked Phase 1 token system and reuse Phase 1/2 primitives wherever possible.

---

## Relationship to Phase 1 + Phase 2

**Phases 1 and 2 are the joint source of truth for all visual tokens.** Phase 3 inherits unchanged:

- **Design system** — hand-rolled RN primitives in `src/components/`, no shadcn (web-only), Manrope variable font, `@expo/vector-icons` (Feather subset).
- **Color palette** — yellow `#FFDE42` primary, cyan `#53CBF3` accent, `hsl(4 78% 56%)` destructive, warm off-white / near-black dual theme. Both themes ship.
- **Typography scale** — Display 32 / H1 24 / H2 20 / Body 16 / Caption 13 (5 sizes — user-locked exception ratified in P1 + P2). Three weights: 500 Medium, 700 Bold, 800 ExtraBold (800 reserved for Display only).
- **Spacing scale** — 4 / 8 / 12 / 16 / 24 / 32 (12px is the user-locked exception from P1).
- **Radii** — 6 (sm) / 12 (md) / 20 (lg) / pill.
- **Elevations** — `e1-subtle` and `e2-raised`. Dark mode: 1px `--border` over shadow.
- **Phase 1 + 2 primitives** — `ScreenContainer`, `ScreenHeader`, `PrimaryButton`, `SecondaryButton`, `GhostButton`, `DestructiveTextButton`, `Modal`, `SegmentedControl`, `TextInput`, `FormLabel`, `FormError`, `Avatar` / `AvatarInitials`, `InviteCodeChip`, `Logo`.
- **Voice rules** — *we / your group / friends* (never *user / organization*); the word `Cancel` is **banned** as a dismiss label — every modal supplies a context-specific dismiss; CTAs are verb + noun.

**Phase 3 adds to, never overwrites, this contract.** The Lovable mockups in `design_refs/design_code_for_claude/accounti-buzz-spark-main/src/mocks/screens/` (Today.tsx, Capture.tsx, Review.tsx) are the **screen-layout source of truth** for this phase; tokens always defer to Phase 1.

### Reconciliation: Lovable mockup vs. locked Phase 1 tokens

The Lovable export ships its own HSL token sheet in `design_refs/.../src/index.css`. It is essentially identical to Phase 1's `src/theme/tokens.ts`. Two minor token-value divergences exist; **Phase 1 wins** in every case:

| Token | Lovable value | Phase 1 value | Winner |
|-------|---------------|---------------|--------|
| `--primary-foreground` | `hsl(220 18% 10%)` | `hsl(220 15% 15%)` (`primaryFg`) | **Phase 1** — already shipped on P1/P2 buttons; both meet WCAG on yellow. |
| Display line-height | `38px` (Lovable Tailwind) | `37px` (`tokens.ts`) | **Phase 1** — 1px difference is invisible at runtime. |
| Body font-weight on muted helper | Lovable `text-muted-foreground` defaults to whatever utility wins; mocks read `font-medium` (500) consistently | Phase 1 Body = 500 Medium | **Phase 1** — already aligned. |

No other divergences. All Lovable component compositions are tokenised against `--primary`, `--surface`, `--muted`, `--destructive`, `--border`, `--foreground`, `--muted-foreground` — every one has a Phase 1 equivalent.

### Where the Lovable mockups win

- Screen layouts (Today, Capture phases, Admin Review queue including all states: idle, swipe-approve, swipe-reject, reject-reason, empty, loading, error, first-time tooltip)
- Component anatomies for the 5 new primitives (GroupCard, StatusPill, TypeChip, QueueBadge, SwipeCard, plus CaptureControls cluster)
- Copy strings — primarily lifted verbatim from the mocks; supplemented by 03-CONTEXT.md (D-01..D-19) and 03-LOVABLE-PROMPT.md tables for surfaces the mocks don't show (e.g. permission-denied microphone copy, RPC typed-error inline copy)
- Microinteractions where the mocks indicate intent (rubber-band animations, swipe overlays, autoplay-muted video, soft cross-fade on status change)

| Token | Value | New P3 usages |

**Exceptions (additive, P3):**

| Exception | Value | Justification |
|-----------|-------|---------------|
| Tab-bar height | 56pt + bottom safe-area | Standard iOS/Android tab-bar metrics — meets 44×44pt hit target per Apple HIG with margin |
| Camera shutter button | 72pt diameter | Larger-than-touch-target intentional — capture is the most-frequent intentional action on this screen; visual prominence reinforces single-tap commitment |
| Camera shutter inner | 52pt diameter | 4pt outer ring (white) + 16pt clearance on each side; matches Lovable mock |
| Active recording inner-square | 16pt | "Stop recording" affordance morphed from circle to square; matches Lovable mock + native iOS Camera convention |
| Top-bar circular controls (close `×`, flip-camera) | 40pt diameter (32pt visual + 4pt scrim padding each side, 44pt hit slop applied) | Visual lightness over viewfinder while preserving HIG 44pt hit target |
| Card-stack offsets | 6pt translateY for card 2 (97% scale), 12pt translateY for card 3 (94% scale) | Lovable-locked stack visual; not a multiple of 4 because the visual relationship is geometric (3pt steps would feel cramped, 8pt would push card 3 below the fold). Logged as a P3-specific stack-visual exception; **does not** propagate to other token usage. |
| Swipe action-button row internal gap | 12pt | `md` token; equal-weight Reject + Approve buttons sit edge-to-edge with this gap |
| Swipe-overlay icon size | 80pt circle (42pt icon inside) | Visual swipe-feedback convention — same scale as iOS card-style stack apps; large enough to read peripherally during gesture |

All other declared values remain multiples of 4. Phase 1's 12px inclusion remains the user-locked spacing exception.

## Color

**Inherited from Phase 1 / 2 — unchanged palette.** New reserved-for assignments and surface mappings:

### Accent Reserved-For List (P3 extension)

`--accent` (cyan `#53CBF3`) continues to be **links-and-focus only**. P3 additions:

1. "Join with a code" text-link on Today screen empty state (sibling to the `Create a group` PrimaryButton — same role as P2's groups-list empty)
2. **Focused border ring** on the reject-reason TextInput (2px ring per existing TextInput convention)
3. **Focused border ring** on the capture caption TextInput (2px ring; rendered against the dark scrim — accent stays cyan, ring is fully visible)
4. **Active tab indicator** — see exception below

`--accent` is still NEVER used for: primary CTAs, body text, icons-by-default, backgrounds, badges, success confirmations, status pills.

**One named exception for the tab-bar active indicator:**
- The 2px-tall `--primary` (yellow) underline above the active tab icon is the brand-accent recall — it explicitly uses `--primary`, not `--accent`. This keeps the tab bar visually anchored to the brand color without leaking yellow into other navigation surfaces. The active tab's *icon and label* color is `--text` (or `--text-strong` in dark) — not yellow — so the yellow is reserved to a 2px line, ~32pt wide.

### Destructive Reserved-For List (P3 extension)

`--destructive` (red) extension for P3:

1. **StatusPill `rejected` state** — `--destructive` text on `--destructive` at ~12% tint background (rendered as `bg-destructive/10` in the mock; mapped to `colors.destructive` with 15% alpha for RN compatibility — see Token Mapping below). No exclamation marks. Tapping opens the rejection-reason Modal.
2. **CTA `rejected` state** on GroupCard — GhostButton variant with subtle 4px-wide `--destructive` left border accent (Lovable mock spec: `absolute left-0 top-2 bottom-2 w-1 rounded-pill bg-destructive`)
3. **Cutoff hint** — `--destructive` color when minutesLeft < 60; `--destructive` + Caption-700 weight when < 5
4. **Reject button (DestructiveTextButton variant)** in the reject-reason flow committal — uses the existing P1 `DestructiveTextButton` primitive variant (red label, no fill), but wrapped inside the panel as a half-width filled button (see SwipeCard spec) — for visual parity with the Approve PrimaryButton. The fallback `Reject` button below the swipe-stack uses the **outlined** variant (transparent bg + 2px `--text/15` border) per Lovable mock; this is a gentler "discoverability" treatment.
5. **Recording shutter** — when video recording is active, the shutter inner-fill swaps from `--primary` (yellow) to `--destructive` (red), with white square inside. The destructive-color usage here is iOS Camera convention — red = "recording active, tap to stop." NOT a destructive-action signal in the user-task sense; documented as a P3 destructive-color exception so the checker doesn't flag it.
6. **Live REC badge** — small destructive-bg pill at top-center during recording, white "REC" label in Caption-700. Same convention as #5.
7. **Inline error toasts in the swipe queue** — `--destructive` text on `--destructive` at ~15% tint background, "Couldn't save that decision. Try again."
8. **Caption char-counter at-cap state** — turns `--destructive` Caption-500 below 5 remaining or at 140 (matches P2 goal-description pattern)
9. **Camera viewfinder caption-input border at over-cap** — destructive border replaces white/20 border (see ReviewPanel mock at 140/140 state)
10. **Swipe-left red X overlay** — large 80pt circle in `--destructive`, white X icon inside, fades in as user swipes left (visual gesture-feedback, not a destructive-action confirmation)

### Primary Reserved-For List (P3 extension)

`--primary` (yellow `#FFDE42`) continues as the brand-CTA filled-button color and the StatusPill-approved emphasis color. P3 additions:

7. **Tab-bar active indicator** — 2px-tall `--primary` underline above the active tab icon (~32pt wide, top-edge of tab cell)

9. **`Back to group` button** on review-queue empty state — PrimaryButton, max-width 320pt centered

### Success-color usage

The mocks use `hsl(145 60% 45%)` (a slightly more saturated success green) for two surfaces: the swipe-right approve-overlay 80pt circle, and the empty-state checkmark badge on the review queue. Phase 1 already declares `--success` as `hsl(145 55% 42%)`. **Phase 1 wins** — these surfaces use `colors.success` from `tokens.ts`, not the Lovable mock's slightly-different shade. The 5-degree saturation difference is imperceptible at icon scale. Both `--success` surfaces (approve overlay + empty checkmark) get a `--primary-fg` (near-black) icon for AAA contrast.

This is the **first non-toast use of `--success`** in the contract. P1 reserved success for "success toasts" (none shipped); P2 used it for the inline `Copied ✓` color swap. P3 elevates it to a brand visual-cue color for two specific places:

- **Swipe-right approve overlay** (ephemeral; appears only during gesture)
- **Empty review-queue badge** ("All caught up" — celebratory, not informational)

**`--success` is NOT used for**: status pills (approved uses `--primary`), buttons, body text, or backgrounds outside these two surfaces.

### `app/(app)/capture/[groupId].tsx` — Capture flow (NEW)

**Modal-style presentation:** routed via Expo Router but uses `presentationStyle="fullScreenModal"` (iOS) / `animation="slide_from_bottom"` (cross-platform) on the Stack.Screen options. The user feels they're "stepping into" capture, not navigating.

**Permission gate (mounts before viewfinder):**
- On mount: call `useCameraPermissions()`. If `status !== 'granted'`, immediately request via `requestPermission()`.
- If granted: proceed to viewfinder.
- If denied: render the **camera permission denied** screen (full-bleed, see screen-level copy). On `Open Settings` tap: call `Linking.openSettings()`. On `Not now` tap: dismiss back to Today.
- For video groups: **after** camera permission granted, also `useMicrophonePermissions()`. Same flow if denied — render the **microphone permission denied** screen.

**Capture state (post-permission, pre-take):**
- Full-bleed `<CameraView />` (expo-camera SDK 55), facing `back` by default
- `CaptureTopBar` overlay
- `BottomPanel` overlay, vertical stack:
  - Video variant only: `RecordingProgressBar` (idle state — full-empty track, label `0:10`)
  - `Shutter` (variant per group's submission_type)
  - Video-idle hint text (Caption-500 white, `Tap to start · 10 seconds max`)

**Recording state (video only):**
- Tap shutter once → `cameraRef.current.recordAsync({ maxDuration: 10 })` and animate `Shutter` to `video-recording` variant
- Live `REC` badge appears top-center (small destructive pill with white dot + `REC` label)
- `RecordingProgressBar` switches to live mode: progress fills 0 → 1 over 10s, timer label counts down `0:10` → `0:00`
- Auto-stop at 10s OR tap shutter again to stop early — both transition to **review state**
- Hide the `Tap to start` hint once recording begins

**Review state (post-capture):**
- Photo: full-bleed still image of the captured photo (use `expo-image` `<Image source={{ uri }} contentFit="cover" />`)
- Video: full-bleed `<VideoView />` (expo-video) with `player.loop = true`, `player.muted = true` (default), `player.play()` on mount
  - Top-right: 36pt circular `rgba(0,0,0,0.55)` scrim with Feather `volume-x` icon (white) — tap toggles mute (icon swaps to `volume-2`)
  - Top-left: small pill `rgba(0,0,0,0.55)` scrim with Caption-500 white `Looping` label
- `CaptureTopBar` remains overlaid at top
- `ReviewPanel` replaces the `BottomPanel` capture controls

**On Submit success:**
- `router.dismiss()` (or `router.back()`) — return to Today
- Today GroupCard for this group will already have updated to `pending` via TanStack Query optimistic update OR will update within ~250ms when the RPC response patches the cache
- **NO success toast.** The dismissal IS the success signal (per CONTEXT D-discretion).

**On Submit error:**
- `loading=false`, `errorText` populated per the typed-error table above
- Button reverts to enabled state
- User can edit caption, retake, or re-submit
- Network failure path: the upload joins the AsyncStorage queue and a Today QueueBadge takes over the visible state — capture screen still dismisses (does NOT block the user on the camera screen waiting for retry; the queue is the durable state)

**Discard-take flow:**
- Tapping `×` (close) when no take has been captured yet: dismiss directly back to Today
- Tapping `×` after a take exists (review state): show **Discard-take Modal** overlaid on the dimmed viewfinder — title `Discard this take?`, body `You'll lose what you just recorded.`, primary `Discard` (destructive filled), dismiss `Keep recording` (ghost)
- On `Discard`: dismiss back to Today, take is discarded (no queue entry created)
- On `Keep recording`: close modal, return to review state

A11y: viewfinder gets `accessibilityLabel="Camera viewfinder"`; shutter labels per `Shutter` spec; review-state media gets the captured-image alt-text equivalent.
