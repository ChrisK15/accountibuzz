# Cross-AI Plan Review — Phase 03.1

## Gemini Review

This review covers the implementation plans for **Phase 03.1: P3 Polish & Realtime Hardening**, comprising plans **01** (Admin Realtime), **02** (Capture Modal), and **03** (Shutter Redesign).

### 1. Summary
The proposed plans are of exceptional quality, demonstrating a deep understanding of both the current codebase and the specific technical constraints of Expo Router and Supabase Realtime. The phase successfully addresses a critical Realtime drift issue, a navigation UX regression, and a specific user UI complaint with surgical precision. The decision to use distinct channel names per mount point (**RESEARCH Q1**) is a high-signal resolution that prevents a known race condition in `supabase-js` during screen transitions. The testing strategy is comprehensive, leveraging established mock patterns to maintain the project's high bar for automated verification.

### 2. Strengths
- **Technical Rigor on Realtime Lifecycle:** The resolution for the channel naming collision (**RESEARCH Q1**) is proactive and correctly identifies a common pitfall in native-stack transitions where source and destination focus effects briefly overlap.
- **Canonical Navigation Patterns:** Plan 02 correctly employs the nested-layout strategy for modal presentation, which is the idiomatic Expo Router solution for applying Stack options within a Tabs context. This restores the spec without risking the `sceneStyleInterpolator` crash seen in Phase 3.
- **API Stability:** The Shutter redesign (Plan 03) preserves the existing `ShutterProps` interface, ensuring that the visual rewrite has zero impact on the complex state machine in the capture screen.
- **Security Discipline:** Plan 01 maintains a three-layer defense (UI gate, RLS, and RPC server-check) for pending submission metadata, ensuring that non-admin members cannot leak or act on review data.
- **Process Compliance:** The plans integrate perfectly with the JIRA/SCRUM convention mandated in `CLAUDE.md`, including a bootstrap task to identify the shared story for this requirement-less phase.

### 3. Concerns
- **Batch Invalidation Volume (LOW):** While the plan uses `qc.invalidateQueries` (D-03), if an admin rapidly reviews many submissions (e.g., 5 approvals in 10 seconds), this will trigger 5 separate network refetches. At MVP volume (≤10 members), this is negligible, but it should be noted if batch-review features are added in Phase 6.
- **Android Transition Behavior (LOW):** On Android, `presentation: 'modal'` and `animation: 'slide_from_bottom'` can sometimes result in different header/status-bar treatments compared to iOS. Since the app is currently iOS-first for UAT, this is acceptable, but the "tab bar covered" truth should be confirmed on an Android emulator.
- **Rules of Hooks Ordering (LOW):** Plan 01 Task 4 correctly identifies the potential TDZ/Ordering risk with the `isAdmin` computation. The suggested insertion point after line 207 is the safest path.

### 4. Suggestions
- **Pulse Animation Polish:** For the Shutter redesign (D-10), if the red recording circle feels too static despite the ring pulse, consider a subtle scale pulse (1.0 ↔ 1.03) on the inner circle. This is left to "Claude's Discretion" in the plan and is a good fallback.
- **Debounce Refetch (Optional):** If UAT shows flickering during rapid review actions, adding a small debounce to the `qc.invalidateQueries` call within the Realtime handler would be a simple optimization, though likely unnecessary for the initial ship.
- **JIRA Story Naming:** Ensure the story created in Plan 01 Task 1 explicitly mentions "P3 Polish" to keep the audit trail clear for reviewers who may not have the phase context.

### 5. Risk Assessment: LOW
The risks are exceptionally low. All patterns used have direct analogs in the existing codebase, and the plans include robust Wave 0 safety-net tests. The most complex item (Realtime lifecycle) is mitigated by the distinct channel naming strategy.

**Final Verdict: APPROVED.** All three plans are ready for execution. No structural changes required.

## Codex Review

### Summary

The three plans are generally strong: they are scoped tightly to the stated Phase 03.1 goals, lean on existing project patterns, include explicit verification gates, and correctly resolve the Q1 channel-name issue by using per-mount-point Realtime channels. The biggest risks are not implementation complexity, but a few plan-quality issues: some UAT expectations are slightly inaccurate, the JIRA workflow is overrepresented inside implementation plans, Plan 03's tests do not actually verify the visual change, and Plan 01's security language overstates what the client-side admin gate/RLS can guarantee for Realtime payload visibility.

### 03.1-01 — Admin Pending-Review Realtime

**Strengths**

- Correctly mirrors the existing `useGroupFeedRealtime` pattern: `useFocusEffect`, server-side `group_id` filter, client-side narrowing, `invalidateQueries`.
- Good Q1 resolution: `review-queue:{groupId}:badge` and `review-queue:{groupId}:list` is the safer choice than relying on undocumented Supabase channel de-duping.
- Strong unit-test plan for subscription shape, no-subscribe admin gate, pending-set narrowing, and cleanup.
- Good choice to invalidate both `['reviewQueue', groupId]` and `['pendingReviewCount', groupId]` rather than optimistic-prepending incomplete rows.
- Mount-point admin gating is explicit and grep-verifiable.

**Concerns**

- **HIGH:** The security model language is too confident. The client-side admin gate prevents accidental subscription by the app UI, but it is not a true authorization boundary. If `submissions_select_group_members` allows all group members to select submission rows, a non-admin with a modified client may still subscribe to `postgres_changes` for that group. The RPC admin check protects enriched queue fetches, but not necessarily raw Realtime row metadata.
- **MEDIUM:** The `must_haves` truth "originating member's pending count refreshes via the same channel" appears inaccurate. This hook is admin-only and invalidates admin queue/count queries. It does not update a submitter/member-side pending count unless that count is actually one of these two query keys on an admin-mounted screen.
- **MEDIUM:** Tests do not cover role revocation / `isAdmin` changing from true to false while the screen remains focused. The hook likely cleans up via dependency change, but this is worth an explicit rerender test because it is one of the requested edge cases.
- **MEDIUM:** UAT for "Device B submits another fresh submission" may be blocked by one-submission-per-member-per-day constraints. The plan mentions a second test member, but this should be a hard setup requirement.
- **LOW:** No debounce is acceptable for MVP volume, but approving/rejecting several rows rapidly will cause duplicate invalidations. Probably fine, but the plan should acknowledge this as intentional rather than accidental.
- **LOW:** Blocking human JIRA setup in Plan 01 can stall otherwise independent implementation. The process requirement is valid, but it couples execution flow to project management setup.

**Suggestions**

- Add one hook test: render with `groupId='g-1'`, then rerender with `undefined`, and assert `removeChannel` was called.
- Reword security claims: "admin gate prevents app UI subscription; server authorization remains RPC/RLS-dependent; raw Realtime visibility follows existing `submissions` RLS."
- Fix the "originating member pending count" wording unless there is a real member-side query this hook affects.
- Make second-member availability explicit in UAT setup.
- Add a short comment that invalidation storms are acceptable at ≤10 members and no debounce is intentionally included.

**Risk Assessment: MEDIUM**

The implementation pattern is low risk, but Realtime authorization assumptions and role-change lifecycle deserve tightening. The Q1 channel-name resolution is sound and lowers operational risk.

### 03.1-02 — Capture Modal Presentation

**Strengths**

- Correct architectural fix: nested `Stack` under the existing Tabs route is the right Expo Router pattern.
- Keeps parent Tabs registration unchanged, which directly protects against the prior `sceneStyleInterpolator` crash.
- Cleanup targets are precise and include a useful `stackScreen` grep gate.
- Manual UAT covers the important user-facing behavior: animation, gesture dismissal, tab bar coverage, cancel, capture flow regression.

**Concerns**

- **MEDIUM:** `presentation: 'modal'` plus `animation: 'slide_from_bottom'` and `gestureEnabled: false` may behave differently across iOS and Android. The plan mentions Android "if available," but the phase is cross-platform Expo; Android should be a required smoke check unless the MVP test target is explicitly iOS-only.
- **MEDIUM:** Expo Router nested layout behavior can be sensitive to route grouping. The plan says no parent layout changes, which is right, but should include a runtime route-open smoke test before UAT, not just typecheck.
- **LOW:** The line-count acceptance range is brittle. Formatting or unrelated nearby edits could make it fail despite a correct cleanup.
- **LOW:** The plan says the file should be ≤35 lines including comments. That is fine but not materially important.

**Suggestions**

- Require Android smoke UAT or explicitly document "iOS-only for this phase."
- Replace strict line-count checks with semantic checks only: no stale tokens, `Shutter` still present, handlers still present, typecheck/test green.
- Add a dev-server restart instruction to verification, not only the action text, because Expo Router file discovery often requires it.
- Add a quick automated check that `app/(app)/_layout.tsx` does not contain `presentation`, `animation`, or `gestureEnabled` on the Tabs screen.

**Risk Assessment: LOW-MEDIUM**

The fix is straightforward and well-scoped. The main uncertainty is platform-specific modal behavior, not code complexity.

### 03.1-03 — Shutter Visual Rewrite

**Strengths**

- Scope is clean: one component plus a small behavior-preservation test.
- Public API preservation is explicit and well verified.
- The redesign directly addresses the complaint by removing `borderRadius: 0` and nested square geometry.
- Uses existing `Feather` icon vocabulary and avoids new dependencies.
- Preserves the existing pulse and press behavior.

**Concerns**

- **MEDIUM:** The tests do not verify the actual visual change. They pass before the rewrite and only protect accessibility/onPress. Grep checks catch some geometry details, but not whether icons render as expected in React Native test output.
- **MEDIUM:** White icons on `t.colors.primary` may not have sufficient contrast if primary is a light yellow. The plan asserts AA contrast without verifying token values. If the app's primary yellow is bright, white may be low contrast.
- **LOW:** The plan says "RED-first" in metadata/task type, then correctly notes it is green-first. This mismatch can confuse executors and reviewers.
- **LOW:** "No glyph" for recording is clean, but a red filled circle can also read as "record" rather than "stop." Since tapping while recording stops recording, a stop affordance might be clearer. The user wanted less blocky, but not necessarily no stop symbol.
- **LOW:** VoiceOver/TalkBack UAT is useful, but this phase's complaint was visual. It is arguably more than needed.

**Suggestions**

- Rename Task 1 from "RED-first" to "behavior-preservation test."
- Add a lightweight assertion using `UNSAFE_getByType(Feather)` or a suitable testID if the icon mock supports it, to verify `camera` and `video` props.
- Verify actual theme contrast for `#FFFFFF` on `t.colors.primary`; if contrast is weak, use a darker icon token for primary idle states and keep white for destructive red.
- Consider a non-square stop affordance for recording, such as Feather `square` is explicitly bad here, but `pause-circle`/`stop-circle` may also reintroduce the complaint. Default clean red circle is acceptable if UAT confirms it.
- Do not make line-count shrinkage an acceptance criterion.

**Risk Assessment: LOW-MEDIUM**

The implementation is low risk, but visual acceptance depends on actual device rendering and contrast. The automated tests are more regression guards than proof of the UI fix.

### Cross-Plan Assessment

**Strengths**

- Plans are correctly decomposed into mostly disjoint file sets and can run in parallel from a code perspective.
- Phase goals are covered: VERIFICATION item 1, VERIFICATION item 2, and the Shutter complaint all have direct tasks and UAT.
- The distinct Realtime channel naming is the right conservative answer to Q1.
- No schema changes, no new dependencies, and no scope creep into deferred items 3-5.

**Concerns**

- **MEDIUM:** All three plans write to the same `03.1-VERIFICATION.md` and shared JIRA story. Parallel execution is safe for source files, but not for verification-summary/JIRA writes unless coordinated.
- **MEDIUM:** The JIRA steps belong in the execution process, but embedding them deeply in every plan adds noise and creates artificial blockers. A single phase-level process checklist would be cleaner.
- **LOW:** UAT receipt numbering across parallel plans can conflict if plans finish out of order unless the executor coordinates.
- **LOW:** Several grep/line-count acceptance criteria are brittle compared with semantic tests.

**Suggestions**

- Mark implementation tasks as parallelizable, but serialize writes to `03.1-VERIFICATION.md`, summaries, and JIRA.
- Move JIRA setup/update into a shared phase-close/process section, while keeping a brief reminder in each plan.
- Replace line-count gates with semantic grep gates and test/typecheck results.
- Add one phase-level final verification pass: `pnpm test`, `pnpm typecheck`, all CK receipts present, no `stackScreen`, no `borderRadius: 0`, no duplicate Realtime mount names.

**Overall Risk: MEDIUM**

The technical changes are mostly low risk and well patterned. The medium rating comes from Realtime authorization/lifecycle subtleties, platform divergence for modal behavior, and coordination overhead from shared verification/JIRA artifacts. The plans should achieve the phase goals with the suggested tightening.
