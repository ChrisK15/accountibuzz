<decisions>
## Implementation Decisions

### Admin Pending-Review Realtime (D-01..D-04)
- **D-01: New `useReviewQueueRealtime(groupId, groupTimezone)` hook,** mirroring the proven `useGroupFeedRealtime` shape (`src/features/submissions/useGroupFeedRealtime.ts`). One file, parallel structure: `useFocusEffect` + `useCallback` + single `supabase.channel(...).on('postgres_changes', ...).subscribe()`. Lifecycle teardown via `supabase.removeChannel(channel)`. Honors P3 D-13 + Pitfall #11.
- **D-02: Single channel `review-queue:{groupId}` filtered server-side on `group_id=eq.{groupId}`** (the only filter `postgres_changes` supports per P3 D-13). Listens to `INSERT` and `UPDATE` events on `public.submissions`. Client narrows further to events that affect the *pending* set: an INSERT with `status='pending'`, OR an UPDATE where `oldRow.status='pending'` (status flipped away from pending — a review just happened) or `newRow.status='pending'` (defensive, shouldn't happen given immutability triggers). Reads `payload.old` reliably because P4 migration 0008 set `replica identity full` on `submissions`.
- **D-03: Patch strategy = `invalidateQueries` only — no optimistic prepend.** Matches `useGroupFeedRealtime` HIGH #8 reasoning verbatim: the Realtime payload doesn't include the embedded profile fields (`display_name`, `avatar_path`, `profile_updated_at`) that the `get_pending_review_queue` RPC returns, so optimistic prepend would either need a follow-up profile fetch or render an incomplete `SwipeCard`. Invalidate both consumer query keys: `['reviewQueue', groupId]` and `['pendingReviewCount', groupId]`. Acceptable ~200ms refetch jump per the P4 D-21 precedent.
- **D-04: Mounted on BOTH consumers.** The hook gets called inside `app/(app)/groups/[id]/index.tsx` (powers the "(N)" badge via `usePendingReviewCount`) AND inside `app/(app)/groups/[id]/review.tsx` (powers the queue list via `useReviewQueue`). Two mount points, but Supabase Realtime de-duplicates identical channel names — one logical channel per `(admin, groupId)` pair. The second mount is effectively free. Consumers that mount the hook simultaneously (admin opens review queue while group-detail is still in the back stack) get one shared channel. Non-admins must NEVER mount this hook — gate the call site by `currentMember.role === 'admin'`, mirroring the existing badge/route gates.

### Capture Modal Presentation (D-05..D-07)
- **D-05: Nested Stack inside the existing `Tabs.Screen`.** Add a new `app/(app)/capture/_layout.tsx` whose default export is a `<Stack>` with `screenOptions={{ presentation: 'modal', animation: 'slide_from_bottom', gestureEnabled: false }}`. The outer `app/(app)/_layout.tsx` `Tabs.Screen name="capture/[groupId]"` registration stays unchanged (`href: null` is preserved — capture is still tab-bar-hidden, just now backed by an inner Stack). Expo Router picks up the nested layout automatically.
- **D-06: `app/(app)/capture/[groupId].tsx` cleans up the lines 156–170 commentary block** once the nested Stack is in place. The deferred-Stack-options TODO at lines 27–31 also becomes obsolete — remove the comment, keep the file's behavior. No `router.push` / deep-link / back-handler audit needed; the path stays at `/capture/[groupId]`.
- **D-07: Tab bar visibility.** The current `Tabs.Screen` registration with `href: null` already keeps capture off the tab bar. With the nested Stack adding `presentation: 'modal'`, the tab bar is fully covered by the modal sheet — no additional `tabBarStyle: { display: 'none' }` overrides needed. Confirm in UAT.

### Shutter Redesign (D-08..D-10)
- **D-08: Icon-based shutter using `Feather` from `@expo/vector-icons`** — already standard across the codebase (`src/components/StatusPill.tsx`, `TypeChip.tsx`, `SwipeCard.tsx`, `GroupCard.tsx`, `CaptureTopBar.tsx`, `feed/StillToPostAvatarRow.tsx`, `feed/FeedItem.tsx`). No new dep adds. Bundled with `expo`, no separate `package.json` entry.
- **D-09: Three-variant shape, redesigned:**
  - `photo` idle: white outer ring + theme-primary fill + Feather `camera` icon centered (white). The icon makes the affordance literal; eliminates the "blocky" complaint by replacing the inner solid square-with-rounded-corners with an icon glyph.
  - `video-idle`: white outer ring + theme-primary fill + Feather `video` icon centered (white). Same shape vocabulary as `photo` — only the icon changes.
  - `video-recording`: white outer ring + destructive-red filled CIRCLE (no nested square, no inner white center square) + subtle ring pulse retained. Replaces the three-nested-shape stack that the user flagged as "square inside a circle."
- **D-10: Pulse animation tuning.** Keep the current 1.4s loop on the outer ring opacity (0.85 ↔ 1.0). The complaint was the geometry, not the animation. If the new red-filled circle reads visually as static, planning may swap the pulse to a scale-up (1.0 → 1.05) on the inner fill instead — Claude's discretion during implementation.

### Conditional-Item Scope (D-11)
- **D-11: VERIFICATION items 3-5 stay deferred** per their original "only on real-world bug report" guard:
  - Item 3 (swipe-stack ergonomics) — ADM-04 ships approved. No reintroduction.
  - Item 4 (reduce-motion) — Shutter pulse + new modal slide-from-bottom respect the existing `Animated.timing` defaults; no `AccessibilityInfo.isReduceMotionEnabled` audit added in this phase. Revisit if a reduce-motion user reports an issue.
  - Item 5 (slow-3G + JWT mid-upload) — supabase-js auto-refresh remains the canonical path; airplane-mode flow already validated in P3 CK-2. No tooling-dependent UAT added.

### Claude's Discretion
- Exact Feather icon choice if `camera` / `video` don't render well at 32–40pt — may swap to `aperture` / `circle` / `play-circle` during implementation; pick whichever reads cleanest at the final size.
- Whether the channel name is exactly `review-queue:{groupId}` vs. `review:{groupId}` vs. `pending-review:{groupId}` — pick one and stay consistent.
- Whether to debounce rapid invalidations on the queue (e.g., admin processes 5 rejects in a row triggering 5 status-flip events) — probably not needed at MVP volume (≤10 members × 1 submission/day per group); revisit only if a perf issue surfaces.
- Precise icon sizing inside the 72×72 pressable — likely 28–34pt; pick what keeps a comfortable visual gap to the outer ring.
- Whether the `video-recording` filled red circle gets a tiny inner negative-space cutout (e.g., a stop glyph) for clarity — Claude's discretion. Default: keep it a clean circle.
- Whether the new `useReviewQueueRealtime` ships with a Jest test that mocks the channel chain (mirroring the `useGroupFeedRealtime` test setup) or relies on the existing channel-chain mock helper — pick during planning per the P4 mock pattern.

<specifics>
## Specific Ideas

- **Concrete repro for D-01..D-04** (from `03-VERIFICATION.md` line 136 — re-confirmed during P4 CK-05): Device A is admin on group-detail; Device B submits a fresh photo/video via the capture flow. Without this phase's hook, Device A's "Pending Review (N)" badge AND the swipe queue at `/groups/[id]/review` do NOT auto-prepend the new pending row within ~2s — admin must back out and re-enter. With the new hook mounted, the invalidation should land within the same ~200ms envelope as the P4 feed-prepend per `04-UI-SPEC.md` line 783.
- **The Realtime gap was masked during P3 UAT** because CK-5 verified the SUBMITTER's status pill update path (the `useTodaySubmissionRealtime` hook), not the ADMIN's pending-queue ingest path. The two paths share no code today.
- **The capture-modal regression was a known inline-fix during P3 UAT**, not a new bug — commit `298fe64` "Capture screen crash on open" stripped the Stack-only options because they leaked into the Tabs context. The fix in this phase reintroduces them in a context where they're valid.
- **The Shutter complaint is geometric, not animation-based.** The user's exact words: "very blocky, its a square inside a circle which makes it look bad. Find some cool icon we can use maybe." The current `video-recording` variant stacks a 4pt outer ring + 44pt red inner square (with `borderRadius: 0`) + 16pt white center square — three concentric shapes. The new design replaces the inner two with a clean filled circle, and the idle states gain an icon glyph for affordance.
- **Channel de-dup is a Supabase Realtime invariant**, not project-level — two `supabase.channel('review-queue:abc')` calls return the same logical channel until the first removes it. So mounting the hook on both group-detail and review screens does NOT double-bill on subscriptions.

</specifics>
