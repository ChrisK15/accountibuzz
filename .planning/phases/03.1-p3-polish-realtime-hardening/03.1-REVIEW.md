## Summary

The phase 03.1 implementation across 3 plans (Realtime admin review queue, capture nested-Stack modal layout, and Shutter visual rewrite) is **structurally sound** — no Critical bugs or security issues found. The Realtime hook (`useReviewQueueRealtime`) faithfully replicates the established `useGroupFeedRealtime` template, the nested `_layout.tsx` is the canonical Expo Router idiom, and the Shutter rewrite preserves the public API contract.

The principal concerns are **leftover UAT diagnostic code in the capture screen** that was meant to be reverted before Phase 3 closed (a `console.error` and a raw error-message exposure in user-facing copy), **missing test coverage** for the third structural narrowing branch in the new Realtime hook, **a fragile DOM-walk assertion** in the group-detail test that may pass coincidentally, and **inconsistent alpha-color helper usage** that the same phase explicitly fixed elsewhere via `applyAlpha`.

The Stack-modal restoration also leaves several now-dead empty `<>...</>` fragment wrappers in `capture/[groupId].tsx` from the strip-only edit (the planned `{stackScreen}` removal — patterns doc line 348 — left the surrounding fragments behind).

## Warnings

### WR-01: Leftover UAT diagnostic `console.error` in capture submit handler

**File:** `app/(app)/capture/[groupId].tsx:228-232`
**Issue:** The capture screen logs raw submission errors to Metro:
```ts
console.error('[capture submit failed]', { msg, err });
```
The inline comment explicitly states: *"Remove this console.error after Phase 3 closes."* Phase 3 has closed (we are in 03.1). This is a debug artifact that ships to production builds, leaks error envelope shape to anyone with a React Native debugger attached, and creates noisy logs in EAS bundle analytics.
**Fix:** Remove lines 229-232 entirely. If diagnostic capture is still desired, route through a structured logger gated by `__DEV__`:
```ts
if (__DEV__) console.warn('[capture submit failed]', { msg });
```

### WR-02: User-facing default error copy still exposes raw error messages (UAT diagnostic mode)

**File:** `app/(app)/capture/[groupId].tsx:269-273`
**Issue:** The default branch of the typed-error switch surfaces raw mutation messages to users:
```ts
default:
  // UAT diagnostic mode: show the raw message ...
  setErrorText(`Submit failed: ${msg || '(no error message)'}`);
```
The comment says "Revert to the friendly copy after we identify all the real-world error shapes." This violates UI-SPEC §"Submit flow error copy" (lines 388-397) which mandates curated user-friendly text. It can also leak internal RPC error codes / RLS messages to end users.
**Fix:** Restore the friendly fallback per UI-SPEC:
```ts
default:
  setErrorText("Something went wrong. Try again in a sec.");
  if (__DEV__) console.warn('[capture] unmapped submit error', msg);
```

### WR-04: Test coverage missing for the third narrowing branch (`flippedToPending`)

**File:** `tests/submissions/useReviewQueueRealtime.test.tsx`
**Issue:** The hook (`useReviewQueueRealtime.ts:80-83`) defines THREE structural cases that trigger invalidation:
1. `insertedPending` — covered (line 95-110)
2. `flippedFromPending` — covered (line 112-132)
3. `flippedToPending` — **NOT covered**

Case 3 (`oldRow.status='approved'` and `newRow.status='pending'`) is described in the source as *"defensive — shouldn't happen via UI but possible via direct DB write or RPC re-flow"*. The IGNORE test at line 134-154 deliberately exercises a *non*-pending → non-pending transition to confirm the negative path. There's no positive-path assertion that the defensive branch actually fires.
**Fix:** Add a sixth test:
```ts
it('invalidates BOTH on UPDATE where newRow.status=pending and oldRow.status=approved (defensive flip-back-to-pending)', () => {
  const { supabase } = require('../../src/lib/supabase');
  const { qc, wrapper, getHandler } = setupChannelMock(supabase);
  const invSpy = jest.spyOn(qc, 'invalidateQueries');
  const { useReviewQueueRealtime } = require('../../src/features/submissions/useReviewQueueRealtime');

  renderHook(() => useReviewQueueRealtime('g-1', 'badge'), { wrapper });
  const handler = getHandler()!;
  act(() => handler({ old: { status: 'approved' }, new: { status: 'pending', id: 's-1' } }));

  expect(invSpy).toHaveBeenCalledWith({ queryKey: ['reviewQueue', 'g-1'] });
  expect(invSpy).toHaveBeenCalledWith({ queryKey: ['pendingReviewCount', 'g-1'] });
```

### WR-05: Inconsistent alpha-color helper usage — string-concat instead of `applyAlpha`

**File:** `app/(app)/groups/[id]/review.tsx:326`
**Issue:**
```ts
backgroundColor: `${t.colors.destructive}26`,
```
This relies on `t.colors.destructive` being a 6-digit hex string. The same phase (`groups/[id]/index.tsx:558`) explicitly fixed an identical bug via the `applyAlpha` helper:
```tsx
// HIGH #5 ... NOT the broken hex-suffix concat that yielded invalid CSS like
// "hsl(220, 14%, 92%)" + the alpha hex byte.
backgroundColor: applyAlpha(t.colors.surfaceMuted, 0.4),
```
If theme tokens ever switch to `hsl()` / `rgb()` (or token plumbing changes during a future P6 polish), this `${...}26` produces invalid CSS like `hsl(0, 80%, 60%)26` and the destructive toast loses its background tint silently.
**Fix:** Import and use the helper:
```ts
import { applyAlpha } from '../../../../src/theme/applyAlpha';
// ...
backgroundColor: applyAlpha(t.colors.destructive, 0.15),
```

### WR-06: D-09 stack-order test relies on incorrect testing-library node traversal

**File:** `tests/groups/groupDetailScreen.test.tsx:393-450`
**Issue:** The test walks the rendered tree to assert Members → PendingReviewRow → destructive-zone ordering, but it traverses `node.children` (the React-fiber/host-node accessor) on testing-library `RenderAPI` instances. Testing-library RN test instances expose `children` as a getter on `ReactTestInstance`, but the implementation here treats them as plain object properties via `node.children` and `Array.isArray(childrenProp)`. The walk also climbs `parent` until finding a `RCTScrollView` / `ScrollView` — but the type comparison `(t as { displayName?: string })?.displayName === 'ScrollView'` will fail for the function-component wrapper `ScrollView` from RN (which is a forwardRef object, not a string and rarely has `.displayName === 'ScrollView'`).

The combined effect: in many cases `walk()` will push nothing into `seen` for one of the three sentinels, the `indexOf` returns -1, and the assertion `expect(idxMembers).toBeGreaterThanOrEqual(0)` would fail. If the test currently passes, it is likely climbing all the way to the root and traversing the entire test renderer tree by accident, which makes the assertion brittle to unrelated reordering.
**Fix:** Use the official testing-library RN traversal (`getAllByType` / `findAllByType`) or capture render output via `getByText(...).parent` chain explicitly. A more robust pattern using accessibility-label ordering:
```ts
import { within } from '@testing-library/react-native';
const { UNSAFE_root } = render(withProviders(<GroupDetailScreen />));
const allText = UNSAFE_root.findAllByType(Text).map((n) => n.props.children).flat();
const idxMembers = allText.findIndex((s) => typeof s === 'string' && /^Members \(/.test(s));
const idxPending = allText.findIndex((s) => typeof s === 'string' && /Pending review \(3\)/.test(s));
const idxDestructive = allText.indexOf('Delete group');
expect(idxMembers).toBeLessThan(idxPending);
expect(idxPending).toBeLessThan(idxDestructive);
```

### WR-07: Dead empty fragment wrappers from the strip-only `stackScreen` removal

**File:** `app/(app)/capture/[groupId].tsx:163-179, 182-193, 195-207, 321-440, 444-547`
**Issue:** Plan 03.1-02 / patterns doc line 346 instructed: *"Delete each `{stackScreen}` occurrence (5 total). Each sits as the first child inside a `<>...</>` fragment opener that is preserved."* The result is several `return <>...</>` blocks that now contain a single child (a `<ScreenContainer>`, `<PermissionDeniedScreen>`, or top-level `<View>`). Empty React fragments wrapping a single child are dead syntax — no behavioral effect, but they obscure the screen's branching structure and complicate future edits (a reader has to verify the fragment is intentional vs. legacy).

Example at lines 163-179:
```tsx
return (
  <>
    <ScreenContainer>
      <View ...><ActivityIndicator ... /></View>
    </ScreenContainer>
  </>
);
```
**Fix:** Remove the now-redundant fragment wrappers. Five cleanups, all identical pattern:
```tsx
return (
  <ScreenContainer>
    <View ...><ActivityIndicator ... /></View>
  </ScreenContainer>
);
```

## Info

### IN-01: `flippedToPending` redundant truthiness check

**File:** `src/features/submissions/useReviewQueueRealtime.ts:80-83`
**Issue:**
```ts
const flippedToPending =
  oldRow?.status &&
  oldRow.status !== 'pending' &&
  newRow?.status === 'pending';
```
The leading `oldRow?.status &&` is redundant — if it's truthy and not equal to `'pending'`, the second clause already filters out undefined/null. The expression evaluates to the falsy `oldRow?.status` value (e.g., `undefined`) rather than `false` when oldRow is missing, which then short-circuits the OR check at line 85 correctly but yields a non-boolean local. Mostly stylistic.
**Fix:**
```ts
const flippedToPending =
  oldRow?.status !== undefined &&
  oldRow.status !== 'pending' &&
  newRow?.status === 'pending';
```

### IN-03: `cutoffLabelFor` accepts an unused parameter

**File:** `app/(app)/groups/[id]/index.tsx:101-103`
**Issue:** `cutoffLabelFor(_groupTimezone: string): string` takes a parameter it never uses. The underscore prefix exempts it from `no-unused-vars`, and the comment block above explains why (MVP cutoff is structurally always midnight). However, the function signature implies tz-dependent behavior that doesn't exist — future callers may pass tz expecting the label to vary.
**Fix:** Either drop the parameter (`cutoffLabelFor(): string`) since the call sites already know the contract, or add an `// eslint-disable-line @typescript-eslint/no-unused-vars` plus a runtime invariant comment. Keeping the parameter is defensible *if* the goal is a stable API for the future per-group-cutoff feature flagged in the comment.

### IN-04: Haptics promise rejection inside outer try/catch could surface a haptics issue as a save error

**File:** `app/(app)/groups/[id]/review.tsx:131-149`
**Issue:**
```ts
try {
  await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success).catch(() => {});
  await reviewMutation.mutateAsync({ ... });
} catch {
  setErrorText("Couldn't save that decision. Try again.");
}
```
`.catch(() => {})` on the Haptics promise correctly swallows haptic failures, but if `Haptics.notificationAsync` throws *synchronously* (e.g., on a device where the native module isn't linked), the throw escapes the `.catch` and lands in the outer `catch`, surfacing a haptic-init error as "Couldn't save that decision." The user has not, in fact, lost their decision.
**Fix:** Wrap the haptic in its own try/catch so it can never poison the mutation flow:
```ts
try { await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success); }
catch { /* haptics unsupported */ }
await reviewMutation.mutateAsync({ ... });
```
