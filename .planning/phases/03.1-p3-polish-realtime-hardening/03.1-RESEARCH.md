**Researched:** 2026-05-09

## Summary

This is a tight three-item polish phase. All three items have established codebase precedent — the planner does NOT have to invent patterns:

1. **Capture modal presentation** — `expo-router` officially supports nesting a `Stack` inside a `Tabs.Screen` to apply Stack-only options (`presentation: 'modal'`, `animation: 'slide_from_bottom'`, `gestureEnabled: false`) that the Tabs navigator type rejects. Add `app/(app)/capture/_layout.tsx`, strip the stale `stackScreen: null` deferral commentary in `app/(app)/capture/[groupId].tsx` lines 27–31 and 156–170.
2. **Admin pending-review Realtime hardening** — `src/features/submissions/useGroupFeedRealtime.ts` is a 1:1 template. Same `useFocusEffect` + `supabase.channel().on('postgres_changes', ...).subscribe()` shape, same `qc.invalidateQueries` patch strategy, same teardown discipline. The new hook narrows on status-flip-touches-pending instead of status-flip-touches-approved, and invalidates `['reviewQueue', groupId]` + `['pendingReviewCount', groupId]` instead of `['groupFeed', ...]` + `['groupLeaderboard', ...]`.
3. **Shutter redesign** — `Feather` glyphs from `@expo/vector-icons@15.1.1` (already used in 8+ components, bundled with `expo`, no dep adds) replace the inner-square geometry. The 1.4s outer-ring opacity pulse is preserved.

**Primary recommendation:** Two PLANs total — one combined plan for the Realtime hook + capture nested layout (both touch core navigation/data plumbing), and one plan for Shutter (pure visual rewrite with no upstream dependency). Plans run in parallel — Shutter has zero file overlap with the other two items.

**HIGH-priority caveat for the planner (Q1 below):** The CONTEXT.md assumption that "two `supabase.channel('review-queue:abc')` calls return the same logical channel" is **NOT verified by Supabase docs** and is contradicted by community discussion (`tried to subscribe multiple times` / `'subscribe' can only be called a single time per channel instance`). Each `.channel()` call creates a fresh client-side channel object. With `useFocusEffect` lifecycle, only one of the two mount points (group-detail OR review-screen) is focused at a time — but the brief overlap during native-stack push transitions can produce duplicate subscriptions. Two viable resolutions; the planner picks one. Both are simple. Default recommendation: distinct channel names per mount point.

### Pattern 1: Nested Stack inside Tabs.Screen for Modal Presentation

**What:** When a route registered inside a `Tabs` layout needs Stack-only options (`presentation`, `animation`, `gestureEnabled`), introduce a nested `Stack` layout file at the route's directory. The Tabs layout dispatches to the nested Stack, which then applies the Stack-only options to the screen file inside.

**When to use:** Any time a screen registered as a `Tabs.Screen` needs modal presentation, slide-from-bottom animation, or gesture-disabled behavior. The Tabs navigator type rejects these options at compile time (this is the source of the P3 UAT `sceneStyleInterpolator` runtime crash, commit `298fe64`).

**Verification:** [CITED: docs.expo.dev/router/advanced/modals/] confirms `Stack` layouts can be nested inside Tabs layouts and apply `presentation: 'modal'` per-screen. [CITED: docs.expo.dev/router/advanced/stack/] confirms `'modal'`, `'slide_from_bottom'`, and `gestureEnabled: false` are all valid SDK 55 Stack options.

**Example:**
```tsx
// app/(app)/capture/_layout.tsx — NEW FILE
// Source: docs.expo.dev/router/advanced/modals/ pattern
import { Stack } from 'expo-router';

export default function CaptureLayout() {
  return (
    <Stack
      screenOptions={{
        presentation: 'modal',
        animation: 'slide_from_bottom',
        gestureEnabled: false,
        headerShown: false,
      }}
    />
  );
}
```

The outer Tabs registration (`app/(app)/_layout.tsx`) stays unchanged at `<Tabs.Screen name="capture/[groupId]" options={{ href: null }} />`. The nested layout takes precedence for the screen file at `app/(app)/capture/[groupId].tsx`.

### Pattern 2: Channel-per-Screen Realtime Hook with `useFocusEffect` Lifecycle

**What:** Subscribe a single Supabase Realtime channel scoped to one screen via `useFocusEffect` — runs on focus, cleans up on blur. The cleanup function calls `supabase.removeChannel(channel)`. Server-side filter is the only filter `postgres_changes` supports (single column equality). Client-side narrowing inside the handler is the second filter.

**When to use:** Any time a screen-bound TanStack Query cache needs Realtime patches. This is the canonical pattern across `useTodaySubmissionRealtime`, `useGroupFeedRealtime`, `useGroupLeaderboardRealtime`, `useGroupTodayCardRealtime` — all four established hooks in the codebase.

**Verification:** [VERIFIED: codebase grep — `src/features/submissions/useGroupFeedRealtime.ts` lines 47–117 + `src/features/submissions/useTodaySubmissionRealtime.ts` line 48 + `src/features/groups/useGroupLeaderboardRealtime.ts` line 53 + `src/features/groups/useGroupTodayCardRealtime.ts` line 39]

**Example (the new hook for this phase):**
```ts
// src/features/submissions/useReviewQueueRealtime.ts — NEW FILE
// 1:1 shape mirror of src/features/submissions/useGroupFeedRealtime.ts
//
// Channel narrows server-side to group_id; client-side narrows to status flips
// touching the `pending` set. Reads payload.old reliably because
// 20260508233129_phase4_points_streaks_feed.sql set replica identity full on
// public.submissions.
//
// Patch via invalidateQueries (NOT optimistic prepend) — same reasoning as
// useGroupFeedRealtime HIGH #8: the Realtime payload doesn't include the
// embedded profile fields (display_name, avatar_path, profile_updated_at) that
// get_pending_review_queue returns.

import { useCallback } from 'react';
import { useFocusEffect } from 'expo-router';
import { useQueryClient } from '@tanstack/react-query';
import { supabase } from '../../lib/supabase';

export function useReviewQueueRealtime(
  groupId: string | undefined,
  // ⬇ 'mountPoint' added per Q1 resolution — distinct channel name per mount
  //   prevents 'subscribe can only be called a single time per channel instance'
  //   errors during native-stack push transitions where both screens overlap briefly.
  mountPoint: 'badge' | 'list',
): void {
  const qc = useQueryClient();
  useFocusEffect(
    useCallback(() => {
      if (!groupId) return;

      const channel = supabase
        .channel(`review-queue:${groupId}:${mountPoint}`)
        .on(
          'postgres_changes' as never,
          {
            event: '*',
            schema: 'public',
            table: 'submissions',
            filter: `group_id=eq.${groupId}`,
          } as never,
          (payload: { new?: unknown; old?: unknown }) => {
            const newRow = payload.new as
              | { status: 'pending' | 'approved' | 'rejected'; id: string }
              | undefined;
            const oldRow = payload.old as
              | { status?: 'pending' | 'approved' | 'rejected' }
              | undefined;

            // Narrow to events that affect the *pending* set:
            //   1. INSERT with status='pending' (new submission entered the queue)
            //   2. UPDATE where oldRow.status === 'pending'
            //      (status flipped away from pending — a review just happened)
            //   3. UPDATE where newRow.status === 'pending'
            //      (defensive — shouldn't happen given 0003 admin-immutable trigger,
            //       but guards against a future migration that allows it)
            const insertedPending = !oldRow && newRow?.status === 'pending';
            const flippedFromPending =
              oldRow?.status === 'pending' && newRow?.status !== 'pending';
            const flippedToPending =
              oldRow?.status && oldRow.status !== 'pending' &&
              newRow?.status === 'pending';
            if (!insertedPending && !flippedFromPending && !flippedToPending) {
              return;
            }

            // Patch — invalidate both consumers of the (admin, groupId) pair.
            qc.invalidateQueries({ queryKey: ['reviewQueue', groupId] });
            qc.invalidateQueries({ queryKey: ['pendingReviewCount', groupId] });
          },
        )
        .subscribe();

      return () => {
        supabase.removeChannel(channel);
      };
    }, [groupId, mountPoint, qc]),
  );
}
```

**Mount points:**
```tsx
// app/(app)/groups/[id]/index.tsx — admin-gated
// (computed `isAdmin = !!user && !!group && group.admin_user_id === user.id`)
useReviewQueueRealtime(isAdmin ? id : undefined, 'badge');

// app/(app)/groups/[id]/review.tsx — route already gates non-admins
useReviewQueueRealtime(isAdmin ? groupId : undefined, 'list');
```

### Anti-Patterns to Avoid

- **Don't introduce a global `<Stack>` at `app/_layout.tsx` to "fix" the modal presentation.** The existing root layout is already a stack at `app/_layout.tsx`, but the capture route is registered inside the `(app)` group whose own layout is `Tabs`. Restructuring the routing tree to elevate capture to the root Stack would touch every route's path resolution and break deep links — the nested-Stack pattern is the canonical fix per Expo Router docs.
- **Don't share an identical channel name across the two mount points.** See Q1 below — `supabase-js` raises `'subscribe' can only be called a single time per channel instance` when two `.subscribe()` calls overlap on the same name. Use `:badge` / `:list` suffixes (or any other distinguisher) per mount.
- **Don't mount `useReviewQueueRealtime` without an admin gate on group-detail.** Non-admins on group-detail must NEVER mount the hook. The RPC's server gate (`get_pending_review_queue` raises `not_admin`) is the underlying enforcement, but the hook fires the channel subscription which is wasteful for non-admins and could log a security advisory if the channel filter ever expanded. The gate is `isAdmin ? id : undefined` at the call site (early-return inside the hook handles `undefined`).
- **Don't optimistic-prepend the Realtime payload into the queue cache.** The payload doesn't include `display_name` / `avatar_path` / `profile_updated_at` — the queue would render an incomplete `SwipeCard` until the refetch lands. This is the same trap `useGroupFeedRealtime` HIGH #8 documented and avoided. Invalidate, don't prepend.
- **Don't add `presentation: 'fullScreenModal'` to the nested Stack.** The original UI-SPEC line 786 said `fullScreenModal`, but `'modal'` (D-05) is the correct default — it's the standard iOS sheet presentation. `fullScreenModal` skips the sheet animation entirely and is visually wrong for the capture flow. The `slide_from_bottom` animation pairs with `'modal'`, not `'fullScreenModal'`.

## Common Pitfalls

### Pitfall 1: Forgetting to remove the inert `stackScreen: React.ReactNode = null` const after introducing the nested layout

**What goes wrong:** The capture screen file at `app/(app)/capture/[groupId].tsx` line 168 declares `const stackScreen: React.ReactNode = null;` and then references `{stackScreen}` in five JSX fragments (lines 186, 206, 221, 346, 470). After the nested layout is added, those references are inert (the const is `null`) and the lines 156–170 commentary describing why they exist is stale.

**Why it happens:** The original P3 inline-fix (commit `298fe64`) replaced a real `<Stack.Screen options={...} />` JSX node with `null`-typed inert variable to keep the JSX shape valid while ripping out the Stack-only options. After the nested layout takes over the modal presentation, the inert const is unnecessary scaffolding.

**How to avoid:** The cleanup task is two-part: (a) drop the `const stackScreen` declaration at line 168 (and the comment block at lines 156–170 explaining it); (b) drop the `{stackScreen}` JSX references at lines 186, 206, 221, 346, 470. The lines 27–31 comment block at the top of the file (the deferred-Stack-options TODO) becomes equally obsolete — drop it. Net result: the file becomes ~25 lines shorter and the modal-presentation comment confusion is gone.

**Warning signs:** A grep for `stackScreen` in the file after the nested layout lands should return zero hits. If references remain, the cleanup was incomplete.

### Pitfall 2: Missing `payload.old.status` on UPDATE (replica identity ≠ full)

**What goes wrong:** The new hook narrows on `oldRow?.status === 'pending'` to detect a status flip away from pending. If `submissions` doesn't have `replica identity full`, `payload.old` only contains the primary key — `oldRow.status` is `undefined`, and the narrowing condition is `false` for every event, so the hook never invalidates on review actions.

**How to avoid:** **No action needed in this phase** — `submissions` already has `replica identity full` set by P4 migration `20260508233129_phase4_points_streaks_feed.sql` line 91 (`alter table public.submissions replica identity full;`). [VERIFIED: file read of P4 migration line 91]. The new hook inherits this for free. If a future migration ever rewrites the table, the replica identity must be re-asserted — flag for P5/P6 migration review checklists.

**Warning signs:** If the swipe queue doesn't update in real time even after this phase ships, check `SELECT relreplident FROM pg_class WHERE relname = 'submissions';` returns `f` (full), not `d` (default). Default would be the regression.

### Pitfall 3: Realtime publication missing `public.submissions`

**What goes wrong:** If `public.submissions` is not in the `supabase_realtime` publication, the channel subscribes successfully but receives zero events — silently broken (this was the P3 CK-5 hard-gate failure root cause; commit `215984b` fixed it).

**How to avoid:** **No action needed in this phase** — verified shipped. P3 migration `20260506165538_phase3_realtime_publication.sql` ran `alter publication supabase_realtime add table public.submissions` (idempotent). The publication state is consistent with `pg_publication_tables`. Re-verify during UAT by checking the channel actually receives events on a cross-device test (one device submits → other device's queue updates without manual refetch).

**Warning signs:** Channel `subscribed` event fires but no event handler is ever called when a fresh submission lands. Reproduce via Supabase MCP by inserting a test row and watching the Realtime debug surface.

### Pitfall 4: `useEffect` instead of `useFocusEffect` (channel leak on tab/screen blur)

**What goes wrong:** A plain `useEffect` cleanup only runs on component unmount. In a Tabs layout, navigating to a different tab does NOT unmount the screen — the channel stays open, the user pays the bandwidth, the backend pays the connection cost, and after enough screen visits the connection cap is hit. This is Pitfall #11 in `.planning/research/PITFALLS.md`.

**How to avoid:** Use `useFocusEffect` from `expo-router`. The cleanup runs on blur (any time the screen is no longer focused — tab switch, navigation push, etc.). The new `useReviewQueueRealtime` follows the canonical pattern (mirrored from `useGroupFeedRealtime` line 47).

**Warning signs:** Open Chrome DevTools → Network → WS panel during navigation. If the websocket connection count grows on every tab switch without dropping, the cleanup is wrong.

### Pitfall 5: Shutter inner fill clipping the outer ring

**What goes wrong:** The current implementation hard-codes the inner-fill size to 44pt for `video-recording` (line 83 of current `Shutter.tsx`) with the explicit comment "44pt so the square's diagonal (62.2pt) stays inside the outer ring's inner diameter (64pt)." The redesign swaps the square for a clean filled circle, which removes the diagonal-vs-diameter constraint — but the inner circle still must not visually crowd the 4pt ring.

**How to avoid:** A 52pt inner circle + 4pt outer ring on a 72pt pressable leaves a 4pt visual gap (72 = 52 + 4 + 4 + 8, where 8 = 4pt ring + 4pt gap on each side). Keep the inner circle at 52pt for all three variants — uniform geometry across photo / video-idle / video-recording. The Feather glyph sits inside the inner circle at 28–34pt (per Claude's discretion D-09). Pick a size that keeps a comfortable gap from the inner circle's edge.

**Warning signs:** The visual gap between the outer ring and the inner fill looks tighter or different across variants. Should be visually identical across all three.

### Pitfall 6: Two channel subscriptions for the same name producing the "tried to subscribe multiple times" error

**What goes wrong:** See Q1 below for the full diagnosis. CONTEXT.md assumes identical channel names auto-de-duplicate; Supabase community discussion + the supabase-js error message contradict this.

**How to avoid:** Use distinct channel names per mount point — `review-queue:{groupId}:badge` (group-detail) and `review-queue:{groupId}:list` (review-screen). Pass a `mountPoint: 'badge' | 'list'` parameter to the hook (or compute it from the call site). Two channel subscriptions per group is fine — each channel filter is server-side `group_id=eq.{X}`, so the bandwidth is identical to one channel. The cost is two websocket subscriptions, not two snapshot fetches.

**Warning signs:** During native-stack push from group-detail to review-screen, the console logs `'subscribe' can only be called a single time per channel instance` or the second `.subscribe()` call returns a `CHANNEL_ERROR` status. Reproduce by quickly tapping the "Pending review (N)" row and watching the Metro log.

```tsx
// src/components/Shutter.tsx — REWRITTEN

import { useEffect, useRef } from 'react';
import { Animated, Pressable, View } from 'react-native';
import { Feather } from '@expo/vector-icons';
import { useTheme } from '../theme/useTheme';

export interface ShutterProps {
  variant: 'photo' | 'video-idle' | 'video-recording';
  onPress: () => void;
}

const ICON_SIZE = 30;  // 28-34pt range per D-09 — pick a value that keeps a comfortable gap to the 52pt inner fill

export function Shutter({ variant, onPress }: ShutterProps) {
  const t = useTheme();
  const isRecording = variant === 'video-recording';
  const innerColor = isRecording ? t.colors.destructive : t.colors.primary;
  const a11yLabel =
    variant === 'photo'
      ? 'Take photo'
      : variant === 'video-idle'
        ? 'Start recording'
        : 'Stop recording';

  useEffect(() => {
    if (!isRecording) {
      pulseOpacity.setValue(1);
      return;

    const loop = Animated.loop(
      Animated.sequence([
        Animated.timing(pulseOpacity, { toValue: 0.85, duration: 700, useNativeDriver: true }),
        Animated.timing(pulseOpacity, { toValue: 1.0, duration: 700, useNativeDriver: true }),
      ]),
    );
    loop.start();
    return () => loop.stop();
  }, [isRecording, pulseOpacity]);

  return (
    <Pressable
      accessibilityRole="button"
      accessibilityLabel={a11yLabel}
      onPress={onPress}
      style={({ pressed }) => ({
        width: 72, height: 72, borderRadius: 36,
        alignItems: 'center', justifyContent: 'center',
        transform: [{ scale: pressed ? (isRecording ? 0.95 : 0.92) : 1 }],
      })}
    >
      {/* Outer 4pt white ring — pulses while recording */}
      <Animated.View
        style={{
          position: 'absolute',
          width: 72, height: 72, borderRadius: 36,
          borderWidth: 4, borderColor: '#FFFFFF',
          opacity: pulseOpacity,
        }}
      />
      {/* Inner 52pt circle — primary yellow OR destructive red.
          NOTE: clean filled circle in all three variants — no nested square,
          no inner white center square (D-09 supersedes UI-SPEC §6b). */}
      <View
        style={{
          width: 52, height: 52, borderRadius: 26,
          backgroundColor: innerColor,
          alignItems: 'center', justifyContent: 'center',
        }}
      >
        {/* Feather glyph for idle states; nothing for video-recording (clean circle). */}
        {variant === 'photo' && (
          <Feather name="camera" size={ICON_SIZE} color="#FFFFFF" />
        )}
        {variant === 'video-idle' && (
          <Feather name="video" size={ICON_SIZE} color="#FFFFFF" />
        )}
      </View>
    </Pressable>
  );
}
```

## Open Questions

### Q1 (HIGH PRIORITY — must be resolved during planning): Channel naming strategy for the two mount points

**What we know:** D-04 mounts the new hook on BOTH `app/(app)/groups/[id]/index.tsx` (admin-gated) and `app/(app)/groups/[id]/review.tsx`. CONTEXT.md `<specifics>` claims: *"Channel de-dup is a Supabase Realtime invariant, not project-level — two `supabase.channel('review-queue:abc')` calls return the same logical channel until the first removes it."*

**What's unclear:** The CONTEXT.md claim is **NOT verified by Supabase docs** and is **contradicted by community discussion**:

- Community discussion [github.com/orgs/supabase/discussions/27142] says: *"When paging occurs a new channel connection starts with the same name while the old one is being unsubscribed. The two realtime websocket calls overlap and the new connection gets messed up."* The recommended workaround is *"avoid sharing the same channel name across multiple components."*
- supabase-js raises an explicit error [VERIFIED: drdroid.io/stack-diagnosis/supabase-realtime-channel-already-subscribed]: *"'subscribe' can only be called a single time per channel instance"* / *"tried to subscribe multiple times."* Each `.channel()` call creates a fresh client-side channel object — there is no auto-de-dup at the supabase-js layer.
- `useFocusEffect` cleanup runs on blur — but during a native-stack push from group-detail to review-screen, the destination's mount fires BEFORE the source's blur cleanup [CITED: reactnavigation.org/docs/navigation-lifecycle/]. There is a brief window where both `useFocusEffect` callbacks have run their setup but neither has run cleanup. If both call `supabase.channel('review-queue:{groupId}')`, the second `.subscribe()` errors out.

**Recommendation (default):** Use distinct channel names per mount point — `review-queue:{groupId}:badge` (group-detail) and `review-queue:{groupId}:list` (review-screen). Pass a `mountPoint: 'badge' | 'list'` parameter to `useReviewQueueRealtime`. Two channel subscriptions per group is fine at MVP volume — each channel filter is server-side `group_id=eq.{X}`, so the bandwidth is identical and the cost is two websocket subscriptions vs. one (negligible).

**Alternative:** Implement an in-process channel-tracker singleton (Set-keyed by name) that tracks active subscriptions and only `.subscribe()` if not already active, with reference-counted teardown. More code, more state, more failure modes — not worth it at this scale.

**Cost of getting it wrong:** During UAT, an admin who taps "Pending review (N)" → swipe-screen → back to group-detail rapidly will see the second subscription error in Metro logs and the badge will stop updating in real time. Same regression as today, just discovered late.

**Resolution:** Planner picks one (default recommended) and documents the decision in the PLAN. The Code Example §2 above assumes the distinct-names path.

However, this phase introduces **zero new write paths, zero schema changes, zero RPC changes, zero RLS policy changes, zero publication changes**. The security surface this phase touches is read-side only (a new channel subscription on an already-published table) plus a UI gate on an admin-only hook. The applicable ASVS categories are narrow:

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|------------------|
| V2 Authentication | no (auth surface unchanged) | — |
| V3 Session Management | no | — |
| V4 Access Control | yes | UI gate at `app/(app)/groups/[id]/index.tsx` (`isAdmin = !!user && !!group && group.admin_user_id === user.id`) before mounting the hook + RLS policy on `public.submissions` (P1 `submissions_select_group_members`) gating the underlying SELECT path the queue RPC uses + server-side `is_group_admin()` precheck inside `get_pending_review_queue` (P3 `20260429173246_phase3_capture_review.sql`). All three layers are already shipped — this phase does NOT add a new layer; it uses the existing three. |
| V5 Input Validation | yes (defense-in-depth) | The new hook treats `payload.new` and `payload.old` as `{...} | undefined` and narrows defensively with optional chaining. Status values are constrained to the `'pending' | 'approved' | 'rejected'` enum at the column level — runtime narrowing in the hook is structural, not validating. |
| V6 Cryptography | no | — |

### Known Threat Patterns for the new hook

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Non-admin subscribes to the channel via raw deep-link or hook misuse, leaking pending submission metadata to a non-admin | Information Disclosure | Three-layer defense already shipped: (1) UI gate at the call site (`isAdmin ? id : undefined`); (2) server-side RLS on `public.submissions` (`submissions_select_group_members`) restricts SELECT to group members regardless of channel — non-admins who ARE in the group will still see Realtime events but cannot act on them (the queue RPC raises `not_admin`); (3) `get_pending_review_queue` RPC raises `not_admin` server-side, so even a non-admin who sees a Realtime event cannot fetch the enriched queue payload. The phase's job is to keep all three layers intact — verify in UAT that a non-admin device on the same group does NOT receive the badge / queue real-time update through this hook. |
| Channel filter mismatch causes cross-group data leak | Information Disclosure / Tampering | Server-side filter is `group_id=eq.${groupId}` — single-column equality is the ONLY filter `postgres_changes` supports per P3 D-13. The string is interpolated from `groupId` (UUID) directly into the filter string, but `postgres_changes` parses the filter as `column=op.value` server-side; injection would require a UUID with `=` or `,` characters, which UUIDs don't contain. Defense-in-depth: the existing pattern across `useTodaySubmissionRealtime` / `useGroupFeedRealtime` / `useGroupLeaderboardRealtime` interpolates `groupId` / `userId` the same way without incident across 4 shipped hooks. |
| Channel leak (unfocused screen continues to receive events) | Resource exhaustion / Privacy | `useFocusEffect` cleanup via `supabase.removeChannel(channel)` on blur. Same pattern as the 4 existing Realtime hooks. Pitfall #11. |
| Two `.subscribe()` calls on identical channel name produce supabase-js error → no real-time updates → silent regression | Denial of service (self-inflicted) | Q1 resolution — distinct channel names per mount point. The error is loud (`'subscribe' can only be called a single time per channel instance` in Metro), so detection is fast in UAT, but resolving it before ship is cheaper. |

**Net result:** This phase introduces no new security surface. The recommended Q1 resolution (distinct channel names) and the admin-gate at the call site are the two security-relevant disciplines.
