## Admin pending-review realtime — how should the new subscription be shaped?

| Option | Description | Selected |
|--------|-------------|----------|
| New `useReviewQueueRealtime` hook, mirror `useGroupFeedRealtime` | Standalone hook mounted on both group-detail (powers `(N)` badge) and review screen (powers queue list). Single channel `review-queue:{groupId}` filtered on `group_id`, client narrows to status flips touching pending. Patch via `invalidateQueries` (consistent with feed pattern; RPC returns enriched profile fields the Realtime payload doesn't have). | ✓ |
| Fold subscription INTO `useReviewQueue` itself | Internal subscription started inside the existing hook so consumers get realtime for free. Simpler API but couples lifecycle to wherever the hook is used and complicates the count-badge case. | |
| Two narrow hooks: one for queue list, one for count badge | `useReviewQueueRealtime` + `usePendingReviewCountRealtime`, each with its own channel. Two subscriptions per admin on group-detail. Cleanest separation but doubles the channel count. | |

**User's choice:** New `useReviewQueueRealtime` hook, mirror `useGroupFeedRealtime`.
**Notes:** Channel-name de-dup at the Supabase Realtime layer means mounting the hook on both group-detail and review screens reuses one logical channel — no double-bill. Admin-only call site at group-detail (gate by `currentMember.role === 'admin'`); review screen mount needs no role gate because the route guard already handles it.

## Record button (Shutter) — what's the actual UI gripe that needs fixing?

| Option | Description | Selected |
|--------|-------------|----------|
| Idle state looks dim/wrong color | Inner fill currently uses theme primary (yellow) on idle. Could swap to white-on-white-ring or a higher-contrast variant. | |
| Sizing / proportion (72×72 outer, 52pt photo / 44pt recording inner) | The 4pt outer ring + small inner fill may look anemic next to camera-app conventions. | |
| Recording state — the red square inside white-square-inside-ring is busy | Three nested squares may be why it looks 'bad'. Could simplify. | |
| Pulse animation timing/feel | 1.4s loop, opacity 0.85→1.0 on the outer ring only. May be too subtle, slow, or animating the wrong element. | |

**User's choice (free-text):** "it looks very blocky, its a square inside a circle which makes it look bad. Find some cool icon we can use maybe"
**Notes:** Geometric complaint, not animation-based. The user wants an icon-based approach instead of nested geometric primitives. This pointed to the follow-up question on which icon vocabulary to use.

## Shutter redesign direction — what target should planning aim for?

| Option | Description | Selected |
|--------|-------------|----------|
| Icon-based: camera/video icons inside a ring | Photo idle: Feather `camera` icon centered in white ring + primary fill. Video idle: Feather `video` icon. Video recording: red filled circle (no nested square) with subtle pulsing ring. Matches the rest of the app's Feather vocabulary. | ✓ |
| Pure circle states, no icon — Apple-Camera style | Photo idle: solid white ring with white fill. Video idle: solid white ring with red fill dot. Video recording: red rounded-square inside the ring, no nested white square. | |
| Custom SVG / illustration shutter | Hand-drawn SVG shutter (e.g. multi-leaf aperture or custom 'go' glyph). Adds `react-native-svg` surface area; not currently in the dep tree. | |
| Defer to Lovable for a Shutter mock first | Drop the question into the per-phase Lovable workflow. Slower but matches the established UI source-of-truth flow. | |

**User's choice:** Icon-based: camera/video icons inside a ring.
**Notes:** Honors the existing Feather vocabulary (8+ components in the codebase). No new deps. The recording-state circle resolves the "square inside a circle" complaint by removing the inner geometric stack entirely.
