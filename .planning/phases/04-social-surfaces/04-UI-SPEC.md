# Phase 4 — UI Design Contract

> Visual and interaction contract for **Social Surfaces** — the leaderboard, group feed, still-to-post avatar row, missed-yesterday tombstones, and the new social-signal line on Today's `GroupCard`. Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Extends `01-UI-SPEC.md`, `02-UI-SPEC.md`, and `03-UI-SPEC.md` — no new visual tokens.** Phase 4 introduces three new RN component primitives (`LeaderboardRow`, `FeedItem`, `StillToPostAvatarRow`) and one inline section pattern (`MissedYesterdayTombstones`), plus a single additive line of content inside the existing `GroupCard` (the social-signal divider line per CONTEXT D-13). All inherit the locked Phase 1 token system and reuse Phase 1/2/3 primitives wherever possible. The Lovable mockup at `design_refs/design_code_for_claude/accounti-buzz-spark-main/src/mocks/screens/Social.tsx` is the **screen-layout source of truth** for this phase.

## Relationship to Phases 1 + 2 + 3

**Phases 1–3 are the joint source of truth for all visual tokens.** Phase 4 inherits unchanged:

- **Design system** — hand-rolled RN primitives in `src/components/`, no shadcn (web-only), Manrope variable font, `@expo/vector-icons` (Feather subset).
- **Color palette** — yellow `#FFDE42` primary, cyan `#53CBF3` accent, `hsl(4 78% 56%)` destructive, `hsl(145 55% 42%)` success, warm off-white / near-black dual theme. Both themes ship.
- **Typography scale** — Display 32 / H1 24 / H2 20 / Body 16 / Caption 13 (5 sizes — user-locked exception ratified in P1 + P2 + P3). Three weights: 500 Medium, 700 Bold, 800 ExtraBold (800 reserved for Display only). One non-weight type stylization carried forward from P3: italic + curly quotes for SwipeCard captions; Phase 4 does NOT re-apply italic to feed captions (see Typography below).
- **Spacing scale** — 4 / 8 / 12 / 16 / 24 / 32 (12px is the user-locked exception from P1).
- **Radii** — 6 (sm) / 12 (md) / 20 (lg) / pill.
- **Elevations** — `e1-subtle` and `e2-raised`. Dark mode: 1px `--border` over shadow.
- **Phase 1 + 2 + 3 primitives** — `ScreenContainer`, `ScreenHeader`, `PrimaryButton`, `SecondaryButton`, `GhostButton`, `DestructiveButton`, `DestructiveTextButton`, `Modal`, `SegmentedControl`, `TextInput`, `FormLabel`, `FormError`, `Avatar` / `AvatarInitials`, `InviteCodeChip`, `Logo`, `GroupCard`, `StatusPill`, `TypeChip`, `SwipeCard`, `Shutter`, `CaptureTopBar`, `ReviewPanel`.
- **Voice rules** — *we / your group / friends* (never *user / organization*); the word `Cancel` is **banned** as a dismiss label — every modal supplies a context-specific dismiss; CTAs are verb + noun.

**Phase 4 adds to, never overwrites, this contract.** The Lovable mockup in `design_refs/design_code_for_claude/accounti-buzz-spark-main/src/mocks/screens/Social.tsx` is the screen-layout source of truth for this phase; tokens always defer to Phase 1.

### Locked Exceptions (named off-grid values, documented so future checkers don't re-flag)

P4 introduces **two** off-grid spacing values. Both follow the named-and-scoped pattern Phase 2 used for typography Locked Exceptions and Phase 3 used for the SwipeCard 6pt offset.

| # | Exception | Standard rule | Actual value | Justification | Scope |
|---|-----------|---------------|--------------|---------------|-------|
| 1 | **Missed-yesterday tombstone avatar diameter** | Avatar diameters in this contract sit at 32 / 36 / 40 / 48 (multiples of 4) | **28pt** (`w-7 h-7`) | The tombstone visual reads as a *softer / smaller* echo of the leaderboard 36pt avatar — explicitly downsized to communicate "demoted, factual, quiet." 32pt would visually equate it with the still-to-post avatar (which is 32pt and represents an active urgency state) — that's the wrong semantic. 28pt is the smallest avatar size that still renders the AvatarInitials text legibly at 13pt caption (verified at design-time). | **Single-use, non-propagating.** Applies only to `MissedYesterdayTombstones` chip-row avatars. Does not propagate to: any other avatar rendering, any future "soft" or "muted" avatar variant, the still-to-post overflow chip (which uses 32pt to match the still-to-post avatars). Future "demoted-state avatar" components must reuse this exception explicitly via `<Avatar size={28} />` or re-justify. |
| 2 | **Section vertical separation `mt-7`** | Section gaps in the inherited token scale are 24pt (`xl`) or 32pt (`2xl`) | **28pt** (Tailwind's `mt-7` = `7 * 4` = 28pt — and Tailwind itself is at 4pt = 1 unit) | This is actually **on-grid** (`7 * 4 = 28`). Lovable mock uses `mt-7` for "Still to post" and "Missed yesterday" section separation, intentionally tighter than `mt-6` (24pt) used for "Today's posts" and "Leaderboard." The visual rhythm is: heavier sections (Leaderboard with 5 rows, Today's posts with multiple cards) get `mt-6`; lighter sections (the single avatar-row Still-to-post and the single tombstone chip-row Missed-yesterday) get the slightly-airier `mt-7` to compensate for their lower vertical content. | **Documented but on-grid.** This entry exists to pre-empt the checker flagging `mt-7` (28) as "not on the 24/32 inherited scale." It IS a multiple of 4. Applies only to "Still to post" and "Missed yesterday" sections inside group-detail. Does not propagate; other future sections continue using `mt-6` (24) or `mt-8` (32). |

**Locked Exception §1 (28pt tombstone avatar) IS off-grid by the avatar-diameter convention** of this contract (which has only emitted 32/36/40/48 to date) but on-grid by the spacing-token rule (`28 = 7 * 4`). The exception documents the **avatar-diameter convention** widening rather than a true off-grid number.

**Locked Exception §2 (28pt section separation) is documented for clarity** because Phase 4 is the first phase to ship a value between `xl` (24) and `2xl` (32) without a token name. No new token added. **Note for future phases:** if a fourth section-separation surface adopts `mt-7`, consider promoting 28 to a tokenised name (e.g. `xl-plus`) in tokens.ts. P4 does not do this; the cost of one named exception in this doc is lower than the cost of widening the token surface mid-project.

The existing Phase 1 user-locked 12px exception and Phase 3 SwipeCard 6pt exception both remain in force unchanged. Phase 4 declares no off-grid number truly outside the multiple-of-4 family.

## Typography

**Inherited from Phases 1 + 2 + 3 — unchanged.** Phase 2's "Locked Exceptions" section (5 sizes / 3 weights from Lovable tokens) carries forward without modification — see `02-UI-SPEC.md §Typography` for the full justification.

### Locked Exception (P4 specific — 800-weight on a non-Display surface)

Phase 1 declared the 800 (ExtraBold) weight as **Display-only** (32pt screen-hero titles). Phase 4's Lovable mock applies `font-extrabold` (800) to the leaderboard points number — rendered at 20pt (Heading 2 size), not 32pt (Display).

| Deviation | Standard rule | Actual value | Justification |
|-----------|---------------|--------------|---------------|
| 800 weight on Heading-2 size | 800 reserved for Display (32pt) only | Heading-2 (20pt) leaderboard points = **20pt size, 800 weight, tabular-nums** | The leaderboard points number is the only surface in P4 with this treatment. Lovable mock specifies it explicitly (`text-h2 tabular-nums font-extrabold`). Justification: the points number is the headline metric of the leaderboard row — it's what users scan first. At 20pt/700 it reads as "another heading"; at 20pt/800 it reads as "a counter." The extra weight is a typographic counter convention (similar to how counters in scoreboard interfaces use heavier weight than surrounding text). Manrope at 20/800 + tabular-nums also dampens digit-jitter when the number changes via Realtime (e.g. 18 → 19 mid-screen). |

**Scope:** Applies only to the leaderboard points number rendered inside `LeaderboardRow`. Does **not** propagate to: section headers (still 20/700), feed item names (still 16/700), member-roster names (still 16/700), or any future scoreboard-style counter. Future counters of this kind must reuse the `LeaderboardRow` points rendering or re-justify this exception explicitly.

## Color

**Inherited from Phases 1–3 — unchanged palette.** New reserved-for assignments and surface mappings:

### Accent Reserved-For List (P4 extension)

`--accent` (cyan `#53CBF3`) continues to be **links-and-focus only**. P4 additions:

1. "See all" / "Show top 5" inline link in the Leaderboard section header (right-aligned, Body-700, `--accent` color)
2. "Show all N members" full-width expand button at the bottom of the Leaderboard card (Body-700, `--accent` color, with leading chevron icon)

`--accent` is still NEVER used for: primary CTAs, body text, icons-by-default, backgrounds, badges, success confirmations, status pills, leaderboard points (which use `--text`), or the social-signal line on Today GroupCard.

### Destructive Reserved-For List (P4 — explicit non-extension)

**Phase 4 adds ZERO new destructive-color surfaces.** This is a deliberate constraint per CONTEXT D-08 ("streak break is rendered quietly"):

- Tombstones use `--surface-muted` at 40% alpha — **never** `--destructive` (no red tombstones, no red banners, no "X just lost their streak" alerts).
- Streak-broken leaderboard rows render with `--text-muted` flame and 0 streak — **not** `--destructive` color.
- The social-signal line's streak-broken variant (e.g. `🔥0`) uses muted color, no flame fade — quiet, not punitive.
- The leaderboard empty-state callout uses `--surface-muted` at 40% alpha — friendly, not error-coded.

**Tone rule — codified:** if a P4 surface communicates "negative member outcome" (missed day, broken streak, zero-rank), the rendering uses `--surface-muted` and `--text-muted`, never `--destructive`. Destructive is reserved exclusively for *user-action consequences* (delete, reject, leave, log out, RPC failure inline error toast) — none of which exist as a P4 user action. The only path that surfaces `--destructive` color in P4 is the Realtime-Realtime patch error fallback (see Realtime Error Handling below), which reuses the P3 inline-error-toast pattern unchanged.

### Primary Reserved-For List (P4 extension)

`--primary` (yellow `#FFDE42`) continues as the brand-CTA filled-button color and the StatusPill-approved emphasis color. P4 additions:

1. **Rank-1 leaderboard chip background** — 24×24 rounded-pill in `--primary` with `--primary-fg` Caption-700 "1" inside. **First non-button, non-StatusPill use of `--primary` since the P2 ADMIN badge.** Justification: the leader-of-the-pack signal is a brand-recall surface; matching ADMIN badge precedent (which used `--primary` for "core trust signal"). Rank-1 is "core social-pressure signal" — a parallel justification.

`--primary` is still NEVER used for: ranks 2 / 3 / 4+ chips (those use `--surface-muted` for ranks 2–3 and no chip for 4+), feed item highlights, leaderboard points number background (the points number itself is `--text` on the row's `--surface` — no fill behind it), or social-signal line emphasis.

### Success-color usage (P4 — none)

Phase 3 elevated `--success` to two surfaces (swipe-approve overlay + empty review-queue badge). Phase 4 adds **zero** new `--success` surfaces. The leaderboard empty-state callout ("Nobody's on the board yet…") explicitly uses `--text-muted` not `--success` because:
- The empty state is *neutral*, not *celebratory* — there's nothing to celebrate yet, just an inviting blank canvas.
- `--success` is reserved for *outcomes* (a swipe approval, a queue cleared); the leaderboard empty state is a *prompt*, not an outcome.

### Realtime status-change copy (no toast, P4 extension of P3)

Phase 3 established that Realtime updates use no toast and no banner — visual cross-fade only. Phase 4 inherits this rule:

- **Leaderboard row Realtime patch** (LB-02): when `points` or `current_streak` changes for a row, the row's affected fragments (`points` number, `streak` number) cross-fade over 250ms (old opacity 1 → 0 in 125ms, new 0 → 1 in 125ms). The whole row does NOT re-shuffle on every update — the parent `useGroupLeaderboard` hook re-sorts the data, and the FlatList animates row position changes via standard `LayoutAnimation` (or no animation if reduce-motion is on). No haptic on Realtime — same rule as P3 ambient pattern.
- **Today's-posts Realtime patch** (FEED-01 + D-21): when a new approval lands, optimistically prepend the FeedItem to the cache. The new item slides in from above over 200ms (`LayoutAnimation.easeInEaseOut`). No toast. No badge. No haptic.
- **Still-to-post Realtime patch** (FEED-02 + D-21 client-narrowing): when a member submits and gets approved, their avatar disappears from the still-to-post row. The avatar fades out over 200ms while the remaining avatars shift left to close the gap. The corresponding FeedItem appears in Today's-posts simultaneously. No toast.
- **Today GroupCard social-signal line Realtime patch** (D-15): when the per-group counter channel reports a change, the line cross-fades the same 250ms (old line opacity 1 → 0 / new line opacity 0 → 1). Per CONTEXT Claude's-Discretion item: animation default is **no animation** (instant text swap) to match the Today screen's calmer rhythm. **DECISION (resolved here):** **Use the instant text swap.** The 250ms cross-fade is reserved for the on-card StatusPill which is the user's own state. The social-signal line is *the group's* state, peripheral by design — animating it would draw attention away from the user's primary CTA. Revisit only if testing shows the instant swap reads as a glitch.
- **Missed-yesterday Realtime patch** — the section never updates within a session; it's read once on focus and only changes at the next midnight rollover (Phase 5). No Realtime needed.

## Screen-by-Screen Contract

### `app/(app)/index.tsx` — Today screen (MODIFIED — additive only)

**Modification:** each rendered `<GroupCard />` now passes a `social` prop (when data available). The GroupCardRow component (the FlatList item that wraps GroupCard with hooks per P3) gains:

- A new hook: `useGroupSocialCounts(groupId)` — composes the per-group Realtime channel (CONTEXT D-15) with three reads:
  - `get_today_posted_count(group_id)` — for the `posted` and `total` numbers (server returns posted; client reads total from existing useGroup data)
  - The user's row from `useGroupLeaderboard(groupId)` — for `points` and `streak`

Defensive: set `numberOfLines={1}` on the points number; ellipsis on overflow is acceptable degradation for the edge case. |
| Tabular-nums on counters | `fontVariant: ['tabular-nums']` on: LeaderboardRow points number, LeaderboardRow streak number, LeaderboardRow rank chip numeral, FeedItem relative time (no — relative time is text-not-counter, no tabular-nums needed), still-to-post `+N` overflow chip, GroupCard social-signal line points and streak fragments. |
